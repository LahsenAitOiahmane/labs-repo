Nature de l’émulateur (AVD sans Play Store)
L’émulateur est un Android Virtual Device (AVD) créé avec Android Studio. C’est une machine virtuelle qui simule parfaitement un vrai smartphone Android sur votre PC (CPU, mémoire, écran, capteurs, etc.).
Pourquoi « sans Play Store » ?

    Quand vous créez un AVD, vous avez le choix entre :
        Images Google Play → Play Store + Google Services préinstallés (bloat, processus en arrière-plan, interférences réseau).
        Images Android pur (ou Google APIs sans Play) → environnement clean, sans aucun service Google.

Avantages pour l’analyse de sécurité (raison pour laquelle on l’utilise ici) :

    Pas de bruit de fond (Google Play Services, Firebase, etc.) → les logs et le trafic réseau sont purs.
    MobSF peut appliquer son proxy HTTPS global sans conflit.
    L’émulateur est non-production et facilement compatible avec le rooting/Frida (MobSF exige un émulateur rooté pour l’analyse dynamique complète).
    Performances optimales sur x86_64 (votre PC).
    Limite : jusqu’à Android API 30 (MobSF ne supporte pas les versions plus récentes pour l’instant car /system n’est plus writable).

MobSF lance automatiquement Frida Server, installe son certificat CA pour intercepter le HTTPS, et capture tout ce que fait l’application en temps réel.

Étape 1 : Création de l’émulateur AVD sans Play Store (10 min)

    Ouvrez Android Studio → Tools → AVD Manager → Create Virtual Device.
    Choisissez un téléphone (ex. : Pixel 5 ou Pixel 6).
    Dans System Image :
        Sélectionnez Android (ou Google APIs) SANS « Google Play ».
        Choisissez API 28 à 30 (recommandé : API 29 ou 30, x86_64).
        Cliquez Download si besoin → Next → Finish.
    Nommez l’AVD : MobSF_DIVA_API_30 (ou similaire).

Vérification : L’image ne contient aucun Play Store. C’est exactement ce qu’on veut.

Étape 3 : Lancement de l’émulateur avec le script MobSF (rooté & prêt pour l’analyse)

    Dans le dossier MobSF :
        Linux / Mac : ./scripts/start_avd.sh
        Windows : scripts\start_avd.ps1
    Le script liste vos AVD → choisissez MobSF_DIVA_API_30 (ou le nom exact).
    Lancez :
    Bash

    ./scripts/start_avd.sh MobSF_DIVA_API_30

    L’émulateur démarre (peut prendre 30-60 secondes la première fois).

Vérification :

    Ouvrez un nouveau terminal et tapez : adb devices
    Vous devez voir : emulator-5554 device (ou emulator-5555).
    Notez cet identifiant (ex. : emulator-5554) → il sera utilisé plus tard.

L’émulateur est maintenant rooté et configuré pour MobSF (le script gère tout).

Étape 4 : Installation et lancement de MobSF via Docker (5 min)
Dans un nouveau terminal (émulateur toujours en marche) :
Bash

docker pull opensecurity/mobile-security-framework-mobsf:latest

Puis lancez MobSF avec la variable d’environnement correcte :
Bash

docker run -it --rm \
  -p 8000:8000 \
  -e MOBSF_ANALYZER_IDENTIFIER=emulator-5554 \
  opensecurity/mobile-security-framework-mobsf:latest

(Remplacez emulator-5554 par votre identifiant exact !)
MobSF démarre. Ouvrez votre navigateur : http://127.0.0.1:8000
Login / Mot de passe par défaut : mobsf / mobsf
Uploaded ImageImportant : L’émulateur doit être lancé AVANT MobSF, sinon l’analyse dynamique échoue.

Gardez le fichier diva.apk (ou DIVA-debug.apk) sur votre bureau.

Étape 6 : Analyse Statique + Dynamique de DIVA (le cœur du lab)

    Dans MobSF → Upload & Analyze → choisissez votre diva.apk.
    Attendez la fin de l’analyse statique (manifest, permissions, code source, etc.).
    Dans le rapport de scan → cliquez sur le gros bouton Dynamic Analysis (ou « Start Dynamic Analyzer »).

Uploaded ImageUploaded ImageMobSF va :

    Installer automatiquement DIVA sur l’émulateur.
    Lancer Frida Server.
    Configurer le proxy HTTPS global.
    Ouvrir l’interface Dynamic Analyzer.

    Dans l’émulateur :
        Lancez l’application DIVA.
        Explorez les 13 challenges (Insecure Logging, Hardcoded, Insecure Storage, etc.).
    Dans MobSF Dynamic Analyzer :
        Runtime Logs → voyez tous les logs de l’application en live.
        Network Traffic → tout le trafic HTTP/HTTPS intercepté (même le SSL !).
        Frida → cliquez « Spawn & Inject » → vous pouvez hooker des méthodes (ex. : Java.perform pour intercepter des appels).
        File Monitor, Intent Monitor, etc.

Exemple concret avec DIVA :

    Allez dans le challenge « Insecure Data Storage » → MobSF capture les fichiers écrits en clair.
    Challenge « Access Control » → voyez les intents non sécurisés.
    Tout est enregistré et exportable en rapport PDF/JSON.
